# Bullet & Summary Library: Knowledge

Patterns for honest, strong CV writing. The owner's actual experience lives in
`references/base_cv.md`, this file holds the *craft*, not the content. As the LEARNING
loop finds phrasings the owner keeps approving, append them (with their real wording)
under "Proven phrasings" below.

## The bullet formula
**Action verb → what you did → tool/skill → result/impact (quantify if real).**

Weak: "Responsible for reports."
Strong: "Built automated SQL reports that replaced a manual weekly process for 3 teams."

## Strong action verbs (vary them)
Built · Automated · Designed · Developed · Implemented · Streamlined · Integrated ·
Analyzed · Optimized · Delivered · Produced · Translated · Collaborated · Shipped ·
Migrated · Reduced · Led (only if true).

## Re-angling the same fact for different roles
One true experience supports different emphases, same facts, different lens:
- Data role lens: lead with the data/SQL/pipeline aspect of the work.
- Backend lens: lead with the systems/API/automation aspect.
- Frontend lens: lead with the user-facing/UI/performance aspect.
Never invent a new aspect, only re-weight what base_cv.md already supports.

## Summary template (3–4 lines, fill from base_cv.md + the JD)
"[Degree/background] with hands-on [skill 1, skill 2] experience from [strongest real
credential: internship/job/freelance]. [One sentence on what they build/deliver].
Seeking a [exact JD role title] role to [honest growth angle tied to a JD skill]."

## Quantify when it's real
Prefer concrete numbers when true ("for N clients", "across 3 departments", "cut
processing from hours to minutes"). If a number isn't known, ask the owner once
(cv_questions) and store the answer in base_cv.md, don't invent.

## Banned
Em dashes · "passionate about" / "I am writing to" · buzzword soup · any tool from the
owner's never-claim tier · vague responsibility-speak ("responsible for", "involved in").
