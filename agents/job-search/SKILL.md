# Job Search Agent
**Agent ID:** job-search
**Schedule:** Every 1 hour
**Purpose:** Find new roles on the owner's platforms, apply all filter rules, create jobCases in Firebase.

---

## Read first
- `_system/CONFIG.md`, machine paths, MCP prefix, timezone. (Everything below assumes
  the MCP server is registered as `jobpilot`; if CONFIG.md says otherwise, use that prefix.)
- `_system/PROFILE.md`, the owner's facts: targets, seniority, work-authorization constraints.
- `_system/INSTRUCTIONS.md`, the owner's UI instructions OVERRIDE the rules below.
- `_system/HEARTBEAT.md`, status reporting (run at start AND end of every run).
- `_system/NOTIFICATIONS.md`, notification schema, priority, dedupe, action links.
- `_system/LEARNING.md`, reflect & improve every run.
- `_system/SCHEMA.md`, the data contract.
- `knowledge/market_playbook.md`, the owner's market: platforms, employers, filters
  (generated by the setup agent, if it's missing, tell the owner to run setup and stop).
- `knowledge/boolean_library.md`, `knowledge/search_strategy.md`, search craft.

**Doc ID = the slug `job-search`** for `agents`, `agentInstructions`, `agentMemory`.

## Step 0: Standard Startup

**0.1 Check if enabled, then START the heartbeat**
```
mcp__jobpilot__get_document("agents", "job-search")
→ If enabled == false: write status:"idle" with lastAction "Disabled, skipped run." and STOP.
→ Capture current runCount (you'll write runCount+1 at the end).
→ Per _system/HEARTBEAT.md, set status:"running", runningSince:<now>, errorMessage:"".
```

**0.2 Read instructions (these OVERRIDE this SKILL, see _system/INSTRUCTIONS.md)**

**0.3 Read memory**
```
mcp__jobpilot__get_document("agentMemory", "job-search")
→ Load searchConfig: targetRoles, targetLocations, seniorityBand, platforms,
  excludeCompanies, remoteOk, lastSearchedAt
→ Load trendingSkills array (carry forward from previous run)
```

**0.4 Check backlog**
```
mcp__jobpilot__list_collection("approvalQueue")
→ Count docs where status == "pending"
→ If count >= 10: write status:"idle" with lastAction "Backlog ≥ 10, paused to avoid
  overwhelming the owner." and STOP cleanly.
```

## Step 1: Search for Jobs

Build searches at runtime from `searchConfig` + `market_playbook.md`. Never hardcode
roles or locations, the owner's targets live in config so they can change them without
editing this file.

### LinkedIn (needs Claude in Chrome, check CONFIG.md connectors)

For each of the 3–4 most relevant combinations of `targetRoles × targetLocations` this
run (rotate combinations across runs so everything gets covered):

```
https://www.linkedin.com/jobs/search/?keywords=<role, URL-encoded>&location=<location>&f_TPR=r86400[&f_E=<seniority codes>]
```
- `f_TPR=r86400` = posted in the last 24 hours (hourly runs only need the freshest)
- `f_E=1%2C2` = entry + associate, apply when `seniorityBand` is intern/junior;
  use `3` (mid-senior) for mid-level owners; omit for senior
- Boolean keyword craft: `knowledge/boolean_library.md`

For each result page, extract: title, company, location (+ remote/hybrid/onsite),
job URL, and a description excerpt big enough to apply the filter rules.

**Limit:** up to 3 pages per search. Stop a search early when every job on a page is
already in jobCases.

### Regional platforms

Use the platforms ranked in `market_playbook.md` (web fetch where the site allows it,
browser otherwise). Same extraction, same limits.

### If browser automation is unavailable
Degrade gracefully: use web fetch for platforms that work without a session, note the
limitation in your run report, and recommend the owner enable Claude in Chrome.

## Step 2: Apply Filter Rules

For EACH job found, apply ALL filters before creating a jobCase. The thresholds come
from PROFILE.md (Targets + Work authorization) and `market_playbook.md`, the examples
below show the shape, not the owner's actual values.

### Experience filter (calibrate to the owner's seniorityBand)
- If the owner is junior: SKIP postings demanding years beyond their experience
  ("3+ years", "minimum 5 years", "senior" in the title unless the body says junior).
  ACCEPT "0–2 years", "fresh graduates welcome", "entry level", "graduate program",
  "trainee", "junior".
- If the owner is mid/senior: invert accordingly, skip internships and roles clearly
  below their band.

### Work-authorization filter (from PROFILE.md + market_playbook.md)
- SKIP postings whose stated citizenship/visa requirements the owner cannot meet
  (in any language the market posts in, the playbook lists the local phrasings).
- ACCEPT "preferred but not required" phrasings, flag them in `searchSummary`.
- When unsure, ACCEPT and flag; job-analysis will dig deeper.

### Role category filter
- ACCEPT: the owner's targetRoles and close variants (the playbook lists market-specific
  title variants worth catching).
- SKIP: categories the owner excluded in PROFILE.md "Avoid", plus obvious mismatches
  (a frontend dev doesn't need warehouse-logistics listings just because they mention
  "JavaScript experience a plus").
- Borderline hybrid roles (e.g. a role that's 50% target / 50% adjacent): ACCEPT with
  `priority: "normal"` and note the split in searchSummary.

### Company filter
- SKIP companies in `searchConfig.excludeCompanies`.
- Mark `priority: "high"` for employers named in `market_playbook.md`'s target list.

```
mcp__jobpilot__list_collection("jobCases")
```

Log duplicates in your run summary; do NOT create a new case.

## Step 4: Create jobCases

For each new job that passed all filters:

```
mcp__jobpilot__add_document("jobCases", {
  "title": "[exact title from JD]",
  "company": "[company name]",
  "location": "[city, country]",
  "workType": "[remote|hybrid|onsite]",
  "jobUrl": "[full URL]",
  "sourcePlatform": "[LinkedIn|<platform>]",
  "deadline": null,
  "salary": null,
  "tags": [],
  "foundAt": "[ISO 8601]",
  "lastUpdatedAt": "[ISO 8601]",
  "status": "found",
  "priority": "[normal|high, high for playbook target employers]",
  "searchSummary": "[see format below]",
  "analysisReport": null, "companyResearch": null, "fitScore": null,
  "atsKeywords": [], "analysisRecommendation": null,
  "cvQuestionsAsked": [], "cvAnswersReceived": {},
  "applicationText": null, "connectionStrategy": null,
  "appliedAt": null, "followUpDate": null,
  "folderPath": "<JOBS_DIR from CONFIG.md>/[Company] - [Title]",
  "driveFolderUrl": null, "driveFileUrls": {},
  "agentLog": [{"timestamp": "[ISO8601]", "agentId": "job-search", "message": "[brief note]"}],
  "errorMessage": null
})
```

After `add_document` returns the Firestore ID, immediately:
```
mcp__jobpilot__update_document("jobCases", "[returned ID]", {"id": "[returned ID]"})
```
(The `id` field must equal the doc ID, every other agent depends on it.)

**searchSummary format:**
```
[Company type/size]. [Role in 1 sentence]. [Experience requirement as stated].
[Work-auth: "no requirement stated" / "requirement X, flagged"]. [remote/hybrid/onsite + city].
```

## Step 5: Update Trends

After all new jobs are processed, update this week's trends doc (doc ID = week key,
e.g. "2026-06-W2", use `set_document` so repeat runs in the same week merge):

```
mcp__jobpilot__set_document("trends", "[YYYY-MM-WN]", {
  "week": "[YYYY-MM-WN]",
  "skillFrequency": {"<skill>": count, ...},
  "roleFrequency": {"<role>": count, ...},
  "companyFrequency": {"<company>": count, ...},
  "certFrequency": {"<cert>": count, ...},
  "jobsFound": [cumulative this week],
  "createdAt": "[ISO 8601]"
})
```

Aggregate skills, roles, companies, and certifications from every JD seen this run
(including filtered ones, they're still market signal). Scan for cert mentions
(cloud certs, vendor certs, language tests, methodology certs). If none: write
`"certFrequency": {}`, don't omit the field. career-advisor depends on this data.

## Step 6: Notify (only if there's signal)

Follow `_system/NOTIFICATIONS.md`. If new jobs were found, write ONE notification:
```
title: "Found [N] new jobs"
body: "[N] new roles queued: [best company - title]. Skipped [total]: [breakdown]."
type: "summary", priority: "normal", actionUrl: "/applications",
dedupeKey: "job-search-found-[YYYY-MM-DD]"
```
If nothing new: **no notification**, report it in systemLogs only.

## Step 9: Reflect & Learn (see `_system/LEARNING.md`)

What worked / was wasteful? What do the owner's recent approvals/rejections reveal about
their taste? Append the signal to `agentMemory`; promote 3×-repeated patterns into
`knowledge/market_playbook.md` (append under "Refresh notes") or `search_strategy.md`.

## Step 10: Close the heartbeat (see `_system/HEARTBEAT.md`)

Write the END heartbeat (status idle, lastRun, nextRun = +60 min, duration, runCount+1,
pendingApprovals, one concrete `lastAction`), and append this run's timestamp to
`agentInstructions/job-search.readBy`.

**Then close every browser tab you opened this run**, keep the owner's browser clean.

---

## Error Handling
- A platform blocks or returns nothing → log it, continue with the next platform.
- Web fetch fails 3× → systemLogs entry + ERROR heartbeat + one `error` notification.
- A single jobCase write fails → skip it, log, continue.
- Never crash silently, always leave a log entry.

## Hard Rules
1. Never create a jobCase that would be immediately rejected by the filter rules.
2. Never skip deduplication.
3. Always set status = "found" on new jobCases, never anything else.
4. Never submit, apply, or contact anyone, search and record only.
5. If approvalQueue backlog ≥ 10, pause and do not create new cases.
